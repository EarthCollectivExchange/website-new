# QLPA Matrix Source

## What This Is

QLPA Matrix Source is a portable, consent-first, local-first, Phi-aligned source foundation for the EarthOS family of systems.

It is not a framework. It is a set of roots — design language, privacy rules, protection architecture, i18n structure, and mode logic — that any EarthOS-aligned project inherits to begin from the same protected, coherent foundation.

---

## Core Intention

Calm protection. Clear consent. Local sovereignty. Truthful language. Privacy by structure. Phi-aligned design rhythm. Multilingual readiness. Reusable intelligence. Transparent maturity. Portable foundation. Life-supporting systems.

---

## QLPA Principles

Nine principles govern every system built on this source:

1. **Truth** — The system only claims what it can actually do. No overclaiming remote deletion, remote protection, or delivery guarantees it cannot verify.

2. **Calm** — Language and interface design reduce anxiety rather than creating urgency. No fear-based copy, no dark patterns, no manufactured urgency.

3. **Consent** — No message, file, voice, or action proceeds without the user's active or clearly standing consent. Consent is never assumed.

4. **Sovereignty** — Users own their data. Local-first by default. Export, clear, and migrate are always available. No silent upload, no silent extraction.

5. **Care** — The system is designed to support human wellbeing, not to optimize engagement metrics. Relationships matter more than session length.

6. **Clarity** — Every action has a visible consequence explained in plain language. What the system does is transparent to the user.

7. **Reversibility where possible** — Actions are reversible where technically feasible. When they are not reversible, this is stated clearly and honestly — never dramatized.

8. **No hidden extraction** — No plaintext message content, file content, voice content, or private keys ever leave the device without an explicit user export action.

9. **No overclaiming** — The system never claims it has deleted data on remote devices when it can only send a request. Certainty levels are always stated accurately.

## QLPA Net Shield Foundation

The Net Shield is a protection architecture with five readiness levels:

| Level | Meaning |
|-------|---------|
| open | No active protection layer |
| protected | Local safeguards active |
| private | No relay, content held locally |
| sealed | Relay-sealed, identity-backed |
| guardian | Guardian network layer (future) |

Shield layers include: Language Shield, Consent Shield, Encryption Shield, Trust Shield, Source Shield, Guardian Shield (future), Mode Shield, Stats Privacy Shield, Recovery Shield (future).

Protection language is always calm and honest. The system never overclaims what a shield level guarantees.

## Language Protocol

The QLPA language protocol defines discouraged terms and their approved replacements.

Examples:

| Discouraged | Approved |
|-------------|---------|
| Panic Mode | Shield Mode |
| Kill switch | Seal Access |
| Emergency wipe | Source Clear |
| Threat detected | New access attempt noticed |
| Delete forever | Clear from this device |
| Disappearing messages | Auto-clear messages |
| Self-destruct | Auto-clear |
| Block user | Close connection |
| Report abuse | Send safety report |

Preferred vocabulary: shield, protect, clear, seal, pause, restore, review, confirm, trust, guardian, source, auto-clear, protection response, delivery paused, held locally, verified, ready, needs review.

## Consent-First Architecture

Every data action has an explicit consent path. Consent is:
- Active, not passive
- Revocable at any time
- Visible in the consent ledger
- Expressed in plain language

---

## Local-First Sovereignty

Local storage is the default for all content. Remote capabilities are scaffold or future, clearly labeled. Nothing is sent outside the device without explicit user action.

---

## Stats Privacy

Stats modes:
- **off** — No stats collected
- **light** — Local aggregate counts only
- **complete** — Richer local analysis (scaffold)

Stats never include: message body, file names, voice content, private keys, contact names, email, phone numbers.

Stats use aggregate classes and size/duration buckets — never personal identifiers.

---

## Active / Scaffold / Future Model

Every capability in the source has a maturity label:

- **active** — Working, tested, live
- **scaffold** — Architecture defined, implementation follows
- **future** — Requires infrastructure not yet built (relay, guardian network, multi-device)

Maturity is always visible to developers. Advanced and developer interface depths surface more maturity context.
